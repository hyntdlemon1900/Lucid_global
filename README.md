## 🏗️ 架构概览 (Architecture Overview)

Lucid-Global 采用**分层调度**设计，将单一集群的精细化管理与多集群间的负载均衡解耦。

### 1. 局部调度器 (Local Scheduler: 原有 Lucid 核心)

每个地理分区（Region/Zone）运行一个独立的 Lucid 实例。它承袭了原论文中**非侵入式**和**抢占敏感**的特性：

* **合并放置策略 (Consolidation Strategy):** 针对深度学习任务对 GPU 拓扑的敏感性（如 8 卡任务必须在同一节点），局部调度器通过高效的 **Bin-packing** 算法，确保任务尽可能紧凑地排列，减少跨节点通信开销并降低资源碎片。
* **公平性与优先级管理:** 基于 Lucid 内部的调度逻辑（如 DRF 或自定义优先级），管理本地等待队列，决定哪些任务优先获取本地算力。
* **细粒度资源抽象:** 不仅仅记录“剩余 GPU 数量”，还实时维护节点内的 **Slot 拓扑图**（哪些机器还剩 1/2/4/8 个连续空闲位置）。

---

### 2. 全局调度器 (Global Scheduler: 本项目新增)

全局调度器充当所有集群的“流量分发中心”。它的核心职责是在任务到达的第一时间，从宏观角度选择最匹配的集群：

* **集群状态聚合 (Status Aggregation):** 全局层定期同步各局部调度器的轻量级快照（Snapshot）。为了保持**可扩展性（Scalable）**，它不监控每一个显存字节，而是监控：
* **资源余量谱 (Resource Spectrum):** 各集群当前能承载的最大连续任务规模（例如：集群 A 剩余 16 卡，但最大连续块仅为 2）。
* **排队压力 (Backlog Pressure):** 局部队列的预估清空时间。


* **智能任务路由 (Job Routing):** 当一个新 Job 提交时，全局调度器计算各集群的**匹配得分**：
* *硬约束检查：* 如果 Job 需要 8 卡，剔除所有无法提供 8 卡连续节点的集群。
* *软目标优化：* 在满足硬约束的集群中，选择 JCT（任务周转时间）最小或数据传输成本最低的路径。


* **负载均衡 (Load Balancing):** 防止出现“一个集群排长队，另一个集群空转”的冷热不均现象，实现地理分布式资源的最优利用。

## 📊 局部调度器上报的统计信息 (Key Metrics)

为了让全局调度器做出准确判断，局部层会周期性上报以下关键数据：

| 统计项 | 物理含义 | 决策用途 |
| --- | --- | --- |
| **GPU Fragmentation Score** | 衡量空闲卡是否分散在不同机器上 | 决定是否能放入大规模并行任务 |
| **Estimated Wait Time (EWT)** | 预测当前队列中新任务需等待的时长 | 最直观的集群繁忙程度反馈 |
| **Resource Histogram** | 统计 1, 2, 4, 8 卡可用槽位的直方图 | 快速筛选符合 Job 硬件需求的集群 |
| **Running Job Residuals** | 正在运行任务的剩余运行时间估计 | 预测近期内会有多少资源被释放 |

## 📈 工作流程 (Workflow)

1. **提交任务:** 用户向 Global Gateway 提交任务需求（如 `nodes: 2, gpus_per_node: 8`）。
2. **全局画像:** Global Scheduler 查看所有局部集群上报的 `Resource Histogram`。
3. **路由决策:** 发现集群 A 虽然空闲多但碎片化严重（无 8 卡节点），而集群 B 虽有排队但即刻有 8 卡节点腾出，决定路由至集群 B。
4. **局部落位:** 集群 B 的 Lucid 实例接收任务，并应用本地合并策略将其放置在最优物理节点。
